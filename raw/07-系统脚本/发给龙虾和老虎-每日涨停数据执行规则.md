# 发给龙虾和老虎：每日涨停数据执行规则

从今天开始，事实层以结果验收为准。龙虾和老虎谁能稳定、完整、准时交付，谁就是主干。

## 一、总原则

1. 所有市场事实先写 RAW，不直接写正式 WIKI。
2. Markdown 给人看，JSON 给 Codex 训练和回测。
3. 每天涨停票必须有一个总入口文件，首板和连板都放在一起。
4. 每日复盘正文只重点分析 2板及以上；首板不逐票铺满正文。
5. 缺数据必须写“数据缺口”，不能编。

老虎还必须先读权限红线：

```text
raw/07-系统脚本/发给老虎-权限边界与禁止改动清单.md
wiki/10-系统配置/老虎事实层权限边界.md
```

老虎只能写事实层 RAW，不能动 WIKI、系统配置、作战室、错误库、模式、统计、脚本和同步配置。

板块字段还必须先读：

```text
raw/07-系统脚本/发给龙虾和老虎-板块字段与板块强度执行规则.md
```

## 二、每天必须交付的 5 组文件

### 1. 每日涨停全景总表

路径：

```text
raw/04-市场数据/每日涨停全景/YYYY-MM-DD/tdx-limit-up-panorama.json
raw/04-市场数据/每日涨停全景/YYYY-MM-DD/tdx-limit-up-panorama.md
```

必须包含：

| 区块 | 要求 |
|---|---|
| 全市场统计 | 剔除 ST 后的涨停总数、首板数、2板及以上数、主板10cm、创业板20cm、科创板20cm、北交所30cm |
| 首板涨停 | 全量首板票，含首板催化 6 要素 |
| 连板天梯 | 剔除 ST 后的全量 2板及以上票，含板数、市场、题材、封板质量 |
| 2板以上6维 | 每只 2板及以上必须有连板 6 维 |
| 板块字段 | 每只股票必须带 `交易题材`、`主板块`、`板块来源`、`概念标签`、`板块归属证据` |
| 数据口径 | ST 直接剔除，不写明细、不单列、不补全；北交所/创业板/科创板不能漏；数据源没覆盖要写清楚 |

### 2. 首板涨停催化

路径：

```text
raw/04-市场数据/首板涨停催化/YYYY-MM-DD/tdx-first-board-catalyst.json
raw/04-市场数据/首板涨停催化/YYYY-MM-DD/tdx-first-board-catalyst.md
```

每只首板必须写“首板催化 6 要素”：

| 字段 | 说明 |
|---|---|
| 题材归属 | 属于哪个题材/产业链，不要只写行业 |
| 事件催化 | 触发涨停的具体消息、公告、政策、涨价、互动易、订单等 |
| 催化来源 | 财联社/公告/研报/互动易/通达信原因/同花顺/公众号/未知 |
| Top100重合 | 是否进入通达信 Top100 或同花顺 Top100，排名多少 |
| 扩散可能 | 是否可能带出同题材补涨、20cm/30cm扩散、产业链扩散 |
| 明日验证点 | 次日竞价、封单、换手、同题材跟随、是否一字/高开低走 |

首板重点是：**这个消息有没有发酵价值，明天有没有补涨/扩散机会。**

每只首板还必须带板块字段：

```text
交易题材
主板块
主板块代码
板块来源
备选板块
行业归属
概念标签
板块归属证据
```

明日验证点必须写事实层可拉数据的条件，不能写“需 Codex 定性”。至少包含：

```text
竞价验证：是否一字、是否高开、是否低于预期
封单验证：封单是否增强、是否撤单、开板后是否快速回封
板块验证：同主板块是否继续涨停、涨停家数是否增加、板块排名是否上升
扩散验证：是否出现 20cm/30cm 扩散，是否有补涨首板
连板验证：同主板块是否有 2板以上带队
热榜验证：是否进入通达信热榜；通达信没有该接口时写“通达信接口无此字段”
消息验证：昨日催化是否有新公告、政策、互动易、研报或财联社继续发酵
```

Codex 再基于这些事实层验证点判断是否升权、降权、进作战室或给交易权限。

### 3. 通达信连板天梯

路径：

```text
raw/04-市场数据/通达信连板天梯/YYYY-MM-DD/tdx-limit-ladder.json
raw/04-市场数据/通达信连板天梯/YYYY-MM-DD/tdx-limit-ladder.md
```

只包含剔除 ST 后的 2板及以上票。

必须覆盖：

```text
主板10cm
创业板20cm
科创板20cm
北交所30cm
```

ST 直接剔除，不写明细、不单列、不补全。

### 4. 通达信涨停原因 6 维

路径：

```text
raw/04-市场数据/通达信涨停原因/YYYY-MM-DD/tdx-limit-reasons-6d.json
raw/04-市场数据/通达信涨停原因/YYYY-MM-DD/tdx-limit-reasons-6d.md
```

每只 2板及以上必须写“连板 6 维”：

| 字段 | 说明 |
|---|---|
| 题材归属 | 属于哪个主线/分支 |
| 事件催化 | 今天继续涨停的直接触发 |
| 产业逻辑 | 这个题材是否有产业链、涨价、订单、政策、业绩支撑 |
| 个股地位 | 龙头/补涨/中军/容量/20cm弹性/30cm弹性/跟风 |
| 盘口质量 | 板型、首次封板、开板次数、封单、成交额、换手 |
| 明日验证点 | 晋级条件、弱转强条件、断板风险、同题材验证 |

2板以上重点是：**市场已经认可后，它还能不能继续接力。**

每只 2板及以上还必须带板块字段，口径同“首板涨停催化”。

### 5. 板块强度

路径：

```text
raw/04-市场数据/板块强度/YYYY-MM-DD/通达信板块强度.json
raw/04-市场数据/板块强度/YYYY-MM-DD/通达信板块强度.md
```

必须包含：

| 区块 | 要求 |
|---|---|
| 板块涨幅全量排名 | 优先全量；接口受限最低Top100；Top50必须写缺口原因 |
| 涨停家数Top20 | 按板块内涨停家数排序 |
| 今日主线候选Top10 | 按涨幅、涨停家数、领涨股、催化强度综合排序 |
| 领涨股 | 每个重点板块至少列3只代表股，能拿多少写多少 |
| 催化摘要 | 政策、涨价、订单、公告、行业事件、外部事件等 |

这个文件用于 Codex 做 D+验证里的“是否跑赢板块”，不能省。

## 三、复盘正文写法

每日复盘正文：

```text
raw/02-每日复盘/YYYY-MM-DD-复盘.md
```

正文只重点分析：

1. 2板及以上连板天梯。
2. 2板及以上逐票 6 维。
3. 首板题材聚类。
4. Top100 重合首板。
5. 最强预期差首板样本。
6. 明日需要验证的 5-10 只重点首板。

不要把全部首板逐票铺满复盘正文。全部首板明细放在“每日涨停全景”和“首板涨停催化”文件里。

## 五、硬性不合格

1. 没有每日涨停全景总表。
2. 只有 Markdown，没有 JSON。
3. 只写 Top10/Top20，不写全量。
4. 首板没有催化来源、Top100重合、明日验证点。
5. 2板以上没有逐票 6 维。
6. 漏掉创业板20cm、科创板20cm、北交所30cm。
7. 把首板混进连板天梯。
8. 数据缺口不写，或者用模糊话糊弄。
9. 个股没有 `主板块` 等板块字段。
10. 没有 `板块强度` JSON/MD。
