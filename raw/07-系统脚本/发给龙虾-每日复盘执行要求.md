# 发给龙虾：每日复盘 RAW 执行要求

龙虾，你以后每日复盘按下面标准执行。你的边界是事实层 RAW：交割单、用户口述、市场事实、热榜、连板天梯、涨停原因、候选事实池。不要直接写 WIKI，不做错误卡最终定性，不升级交易模式，不写作战室最终结论。

硬要求：你写完的交割单和每日复盘必须同步到 Mac。Codex 会把它们正式发布进 WIKI：

```text
raw/01-交割单/YYYY-MM-DD/交割单.md -> wiki/06-持仓与资金管理/YYYY-MM-DD-交割单.md
raw/02-每日复盘/YYYY-MM-DD-复盘.md -> wiki/09-统计与进化/YYYY-MM-DD-复盘.md
```

如果 Mac 端没有收到 RAW 交割单或复盘，系统会判定“每日 WIKI 入库未完成”。

## 一、固定写入路径

```text
raw/01-交割单/YYYY-MM-DD/原始凭证/
raw/01-交割单/YYYY-MM-DD/交割单.md
raw/01-交割单/YYYY-MM-DD/交割单结构化.json
raw/02-每日复盘/YYYY-MM-DD-复盘.md
raw/04-市场数据/通达信热榜/YYYY-MM-DD/tdx-hot-top100.json
raw/04-市场数据/通达信连板天梯/YYYY-MM-DD/tdx-limit-ladder.json
raw/04-市场数据/通达信涨停原因/YYYY-MM-DD/tdx-limit-reasons-6d.json
raw/04-市场数据/首板涨停催化/YYYY-MM-DD/tdx-first-board-catalyst.json
raw/04-市场数据/首板涨停催化/YYYY-MM-DD/tdx-first-board-catalyst.md
```

如果暂时写到旧路径，也必须在复盘里写清楚实际路径。

## 二、复盘头部必须有

```yaml
trade_date: YYYY-MM-DD
writer: longxia
source_type: feishu_oral_review_with_trade_slip
status: RAW 待 Codex 主流程提炼
needs_codex_followup: true
trade_slip:
  - raw/01-交割单/YYYY-MM-DD/交割单.md
user_review_source:
  - 飞书口述
```

缺 `writer: longxia` 或 `needs_codex_followup: true`，Mac 侧 Codex 不会自动接力。

## 三、每天必须交付的内容

1. 交割单：原始凭证、成交摘要、持仓摘要、每笔成交理由。
2. 用户飞书口述：必须保留原话，不要改写，不要替用户编理由。
3. 通达信热榜 Top100：必须全量 100 只，不能只写 Top10/Top20。
4. 连板天梯：只拉取并写入 2板及以上，含主板 10cm、创业板 20cm、科创板 20cm、北交所 30cm；首板不得混入天梯。
5. ST 口径：直接剔除，不写明细、不单列、不补全。
6. 首板涨停催化日报：每天首板涨停票单独拉数据、单独写 RAW/JSON，不进入连板天梯。
7. 涨停原因 6 维度：每只 2板及以上连板票必须拆 6 维；首板在首板催化日报里至少写清涨停原因、催化来源和明日验证点。
8. 财联社/公众号催化：只保留能刺激题材或个股的高信号内容，并标注对应方向/个股。
9. 次日候选事实池：只做事实候选，不给最终买入权限。
10. D+验证任务：核心候选必须写 D+1/D+3/D+5 验证点。
11. 数据缺口：缺什么写什么，不能假装完整。

## 四、连板天梯必须分清口径

每天必须分开写 2板及以上：

```text
连续涨停高度
N日M板高度
反包板
趋势新高板
```

不能把 `3连板` 和 `5天3板` 混在同一列。
首板不是连板天梯的一部分，必须写入独立的首板涨停催化日报。

每天必须有这个口径：

```yaml
连板天梯摘要:
  剔除ST后涨停总数:
  主板10cm数量:
  创业板20cm数量:
  科创板20cm数量:
  北交所30cm数量:
  ST口径: 已剔除，不写明细、不单列、不补全
  连续涨停最高板:
  N日M板最高板:
  2板数量:
  3板数量:
  4板数量:
  5板及以上数量:
  首板数量:
  首板催化日报路径:
  断板数量:
  数据缺口:
```

如果北交所当天没有涨停，必须写：

```yaml
北交所30cm数量: 0
```

如果数据源没覆盖北交所，必须写：

```yaml
数据缺口: 北交所 30cm 数据源未覆盖
```

## 五、涨停原因 6 维度格式

每只 2板及以上连板票至少拆成下面 6 字段：

| 代码 | 名称 | 题材归属 | 事件催化 | 产业逻辑 | 个股地位 | 盘口质量 | 明日验证点 |
|---|---|---|---|---|---|---|---|

盘口质量至少包含：

```text
板型、首次封板、开板次数、封单、成交额、换手
```

缺字段就写“数据缺失”，不要省略。

## 六、JSON 不能污染训练集

Markdown 给人看，JSON 给 Codex 训练和回测。JSON 的全量/样本口径必须写清楚。

Top100 全量 JSON：

```yaml
meta:
  date: YYYY-MM-DD
  total: 100
  是否样本: false
```

连板天梯全量 JSON：

```yaml
meta:
  date: YYYY-MM-DD
  是否样本: false
  选择规则: 剔除 ST 后的全市场 2板及以上连板全量，首板另走首板涨停催化日报
  2板及以上总数:
  剔除ST后2板及以上总数:
  市场范围:
    主板10cm数量:
    创业板20cm数量:
    科创板20cm数量:
    北交所30cm数量:
    ST口径: 已剔除，不写明细、不单列、不补全
    数据缺口:
```

首板涨停催化日报 JSON：

```yaml
meta:
  date: YYYY-MM-DD
  是否样本: false
  选择规则: 剔除 ST 后的全市场首板涨停全量
  首板总数:
  path_md: raw/04-市场数据/首板涨停催化/YYYY-MM-DD/tdx-first-board-catalyst.md
items:
  - code:
    name:
    market:
    limit_pct:
    theme:
    催化来源: 财联社/公告/研报/互动易/通达信原因/未知
    催化原文:
    证据路径:
    热榜排名:
    Top100重合:
    明日验证点:
    数据缺口:
```

如果 JSON 只写核心样本，必须写：

```yaml
meta:
  是否样本: true
  选择规则: 只保留连板/高辨识度/Top100重合样本
  full_source_path: raw/02-每日复盘/YYYY-MM-DD-复盘.md#七、连板天梯与情绪锚
```

禁止出现：`元信息.剔除ST后数量=140`，但 JSON 只有几十只，且不写 `是否样本: true`。

## 七、13 段复盘结构

```text
一、今日一句话结论
二、赚钱效应与市场环境
三、盘前计划回看
四、今日实际交易
五、持仓与资金
六、主线与板块
七、连板天梯与情绪锚
七甲、首板涨停催化日报
七补、通达信热榜 Top100
八、用户口述原文
九、操作质量初步打分
十、错误候选
十一、D+验证任务
十二、明日操盘要点初稿
十三、待用户补充
```

第九、十、十一、十二段只做事实层初稿，最终判断由 Codex 主流程处理。

## 八、禁止偷懒写法

以下都不合格：

1. “今日科技很强，明天继续看科技。”
2. “热榜前排很强。”但不给 Top100。
3. “连板高度不错。”但不列完整天梯。
4. 把首板混在连板天梯里，或者只写“首板很多”但不单独写催化日报。
5. “涨停原因见通达信。”但不拆 6 维度。
6. “明天观察。”但不给具体 D+1 验证条件。
7. 涨停总数、首板数、2板及以上数这类口径不一致，但不解释差异。
8. 把创业板 20cm、科创板 20cm、北交所 30cm 从天梯里过滤掉。
9. JSON 只有几十只核心票，但写得像全量。
10. 改写用户飞书原话，或者替用户编买卖理由。

## 十、Codex 会怎么验收

Mac 侧会自动或手动执行：

```bash
python3 .system/scripts/check-longxia-review-quality.py --date YYYY-MM-DD --write
```

验收不过，会生成：

```text
wiki/09-统计与进化/YYYY-MM-DD-龙虾复盘RAW质量验收.md
wiki/09-统计与进化/YYYY-MM-DD-龙虾复盘补交任务单.md
.system/feishu-notify-pending/YYYY-MM-DD-龙虾复盘质量提醒.md
```

你不需要写 WIKI，但 RAW 必须经得起这个验收。
