# 龙虾每日复盘 RAW 质量验收与偷懒识别规则

更新时间：2026-07-03

## 目标

龙虾负责事实层 RAW。Codex 负责检查龙虾 RAW 是否真的达到短线复盘要求，并把缺项写成可转发给用户和龙虾的通知。

验收不是为了挑毛病，而是为了保证：

1. Top100、连板天梯、涨停原因事实源、财联社、公众号这些短线必看数据没有被省略。
2. 数据口径清楚，能被 Codex 打分、排序、验证。
3. 每日复盘能支持次日涨停预判和 D+验证。
4. 龙虾不使用 DeepSeek 做深度分析，避免重复消耗模型费用；分析由 Codex Pro 完成。

## 龙虾必须交付

每天收盘复盘 RAW 必须包含：

| 模块 | 最低要求 | 偷懒判定 |
|---|---|---|
| 交割单 | 原始凭证、成交摘要、持仓摘要、每笔成交理由 | 只有总盈亏，没有逐笔成交 |
| 用户口述 | 飞书原话完整保留 | 改写用户原话，或只写一句概括 |
| 通达信热榜 Top100 | 全量 100 只，写正文或附录，并尽量写 JSON | 只写 Top10/Top20，或写“热榜很强” |
| 剔除 ST 后的 2板及以上连板天梯 | 全量写入 2板及以上，含主板、创业板 20cm、科创板 20cm、北交所 30cm，区分连续板、N日M板、反包板、趋势新高板 | 只写高标，不写 2板；把首板混进天梯；漏掉 20cm/北交所 |
| 首板涨停事实源 | 首板涨停票单独写 RAW/JSON，至少包含原始涨停原因、来源、板块、Top100 重合、盘口字段 | 只写“首板很多”，或把首板塞进连板天梯 |
| 涨停原因事实源 | 每只 2板及以上连板票必须写原始涨停原因、板型、封单、开板、成交额、换手、来源、主板块 | 把所有原因塞进一句“核心催化”，或用 DeepSeek 写空泛判断代替事实字段 |
| 财联社/公众号催化 | 写原始标题、时间、来源、涉及板块/个股；是否重要由 Codex Pro 判断 | 只贴新闻标题且没有来源/时间/映射标的 |
| 可验证字段 | 写今日一字/高开/封单/竞价/晋级/Top100变化/公告新闻原文等事实 | 只写“明天观察”，没有可验证事实字段 |
| 数据缺口 | 缺什么写什么，不能假装完整 | 口径冲突但不解释 |

## 连板天梯口径

龙虾必须分清四类：

```text
连续涨停高度
N日M板高度
反包板
趋势新高板
```

禁止把 `连续 3 板` 和 `5 天 3 板` 混在同一列里。

每日必须写：

```yaml
连板天梯摘要:
  剔除ST后涨停总数:
  主板10cm数量:
  创业板20cm数量:
  科创板20cm数量:
  北交所30cm数量:
  ST口径: 已剔除，不写明细、不单列、不补全
  2板及以上总数:
  连续涨停最高板:
  N日M板最高板:
  2板数量:
  3板数量:
  4板数量:
  5板及以上数量:
  首板数量:
  首板催化日报路径:
  断板数量:
  数据缺口:
```

北交所和 20cm 票不能被过滤。若当天没有北交所涨停，必须明确写 `北交所30cm数量: 0` 和数据源口径。
首板不属于连板天梯，只能作为数量摘要和独立文件路径出现。

## JSON 全量/样本口径

结构化 JSON 会进入 Codex 训练和回测，不能让“核心样本”伪装成“全市场全量”。

全量 JSON 必须写：

```yaml
meta:
  是否样本: false
  2板及以上总数:
  剔除ST后2板及以上总数:
  选择规则: 剔除 ST 后的全市场 2板及以上连板全量，首板另走首板涨停催化日报
  市场范围:
    主板10cm数量:
    创业板20cm数量:
    科创板20cm数量:
    北交所30cm数量:
    ST口径: 已剔除，不写明细、不单列、不补全
    数据缺口:
```

首板涨停催化日报 JSON 必须写：

```yaml
meta:
  是否样本: false
  首板总数:
  选择规则: 剔除 ST 后的全市场首板涨停全量
```

样本 JSON 必须写：

```yaml
meta:
  是否样本: true
  选择规则: 只保留连板/高辨识度/Top100重合样本
  full_source_path: raw/02-每日复盘/YYYY-MM-DD-复盘.md#七、连板天梯与情绪锚
```

偷懒判定：`元信息.剔除ST后数量` 写 100 多只，但 JSON 只有几十只，且没有 `是否样本: true` 和 `选择规则`。

## 涨停原因事实源

低成本模式下，龙虾不再用 DeepSeek 写 6 维深度分析。每只 2板及以上连板票必须尽量交付原始事实字段：

| 字段 | 说明 |
|---|---|
| 通达信原始涨停原因 | 接口原文，不改写成判断 |
| 题材/概念原始标签 | 接口返回什么写什么，保留来源 |
| 板型 | 一字、T字、换手、炸板回封等 |
| 封单/开板 | 封单金额、开板次数、首次/最终封板时间 |
| 成交额/换手 | 原始数值 |
| 主板块/主板块代码 | 统一中文字段 |
| 来源 | TDX、同花顺、公告、财联社、互动易等 |

Codex Pro 根据这些事实源再生成 6 维分析：题材归属、事件催化、产业逻辑、个股地位、盘口质量、明日验证点。

## 结构化 RAW 要求

Markdown 给人看，JSON 给 Codex 训练和回测。每天应尽量同步写：

```text
raw/04-市场数据/通达信热榜/YYYY-MM-DD/tdx-hot-top100.json
raw/04-市场数据/通达信热榜/YYYY-MM-DD/tdx-hot-top100.md
raw/04-市场数据/同花顺热榜/YYYY-MM-DD/ths-hot-top100.json
raw/04-市场数据/同花顺热榜/YYYY-MM-DD/ths-hot-top100.md
raw/04-市场数据/每日涨停全景/YYYY-MM-DD/tdx-limit-up-panorama.json
raw/04-市场数据/每日涨停全景/YYYY-MM-DD/tdx-limit-up-panorama.md
raw/04-市场数据/通达信连板天梯/YYYY-MM-DD/tdx-limit-ladder.json
raw/04-市场数据/通达信涨停原因/YYYY-MM-DD/tdx-limit-reasons-source.json
raw/04-市场数据/通达信涨停原因/YYYY-MM-DD/tdx-limit-reasons-source.md
raw/04-市场数据/首板涨停催化/YYYY-MM-DD/tdx-first-board-catalyst.json
raw/04-市场数据/首板涨停催化/YYYY-MM-DD/tdx-first-board-catalyst.md
```

如果当天无法生成 JSON，必须在复盘中写 `数据缺口：结构化 JSON 未生成`。

## Codex 自动验收

Mac 侧检查脚本：

```bash
python3 .system/scripts/check-longxia-review-quality.py --date YYYY-MM-DD --write
```

输出：

```text
.system/longxia-review-quality-YYYY-MM-DD.json
wiki/09-统计与进化/YYYY-MM-DD-龙虾复盘RAW质量验收.md
.system/feishu-notify-pending/YYYY-MM-DD-龙虾复盘质量提醒.md
```

严重问题包括：

1. 没有 Top100 全量。
2. 没有首板涨停催化日报。
3. 没有完整 2板及以上连板天梯。
4. 把首板混进连板天梯。
5. 连板口径混乱且无解释，或没有说明 10cm/20cm/30cm 市场范围。
6. 2板及以上连板票没有逐票原始涨停原因、板型、封单、换手、成交额、来源。
7. 缺交割单或用户原话。
8. 没有可用于 Codex Pro 生成明日验证点的事实字段。
9. 结构化 JSON 样本口径不清，可能污染 Codex 训练集。

## 飞书通知口径

如果缺项严重，Codex 生成飞书通知文案，用户可以直接转给龙虾。以后配置飞书 webhook 后，脚本可自动推送。

待发送目录：

```text
.system/feishu-notify-pending/
```

如配置 `FEISHU_WEBHOOK_URL` 或 `LARK_WEBHOOK_URL`，可运行：

```bash
python3 .system/scripts/send-feishu-pending-notifications.py
```

通知只说事实：

```text
龙虾，今天复盘 RAW 有以下硬缺项：
1. ...
2. ...
请补到 raw/02-每日复盘/YYYY-MM-DD-复盘.md，并同步补结构化 RAW。
```

## 一句话

龙虾可以少写判断，但不能少交事实；可以有数据缺口，但必须明说；不能把 Top100、首板事实源、2板及以上连板天梯和逐票涨停原因事实源省掉。深度6维分析由 Codex Pro 完成。
