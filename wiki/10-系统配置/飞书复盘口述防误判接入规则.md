# 飞书复盘口述防误判接入规则

更新时间：2026-07-03

## 目标

用户通过飞书口述复盘、买卖理由、消息判断、校准意见时，系统必须先准确保存原话，再做分析。

核心原则：

```text
先原话 RAW -> 再低风险识别 -> 再歧义回执 -> 最后正式复盘/作战室/WIKI
```

## 统一脚本

飞书桥接器收到文字消息后，优先调用：

```bash
cd /Users/qixinchaye/wiki/73神话
python3 raw/07-系统脚本/codex_ingest_feishu_message.py --stdin
```

指定为复盘口述：

```bash
python3 raw/07-系统脚本/codex_ingest_feishu_message.py --date YYYY-MM-DD --type review --stdin
```

## 连续复盘口述

用户说：

```text
开始复盘
开始复盘 20260703
```

系统进入连续记录模式。

用户后续每段话都追加到：

```text
raw/02-每日复盘/YYYY-MM-DD-飞书复盘RAW.md
raw/02-每日复盘/YYYY-MM-DD-飞书复盘RAW.jsonl
```

用户说：

```text
完成复盘
今天复盘就这么多
今天复盘就这么多，写入Wiki
```

系统退出连续记录模式，并进入复盘送审流程。

## 永远写入的原始沟通

每条飞书交易相关消息都写入：

```text
raw/10-飞书交易沟通/YYYY/MM/DD/
```

这只是原始沟通，不等于短线知识，不等于正式复盘。

## 回执格式

飞书只回短回执：

```text
已按原话写入 RAW。
落点：...
识别：股票/动作/情绪分
待确认：...
我不会把待确认内容写成你的正式买卖理由。
```

不要默认回：

```text
job_id
内部路径长解释
后台日志
大段复盘总结
```

## 误判防线

| 场景 | 处理 |
|---|---|
| 用户说“这个/那个/刚才”但对象不明 | 标 `待确认` |
| 用户说买入/卖出但没有票 | 标 `买卖对象未明确` |
| 用户在问“什么意思/重不重要” | 只作为问题，不当作用户观点 |
| 用户说“可能/感觉/好像” | 作为当时感受，不当作事实 |
| 用户纠正“这条没用/降权/升权” | 写入校准事件，进入后续验证 |
| 用户明确“写成规则” | 先入待验证规则，再看市场验证 |

## 正式复盘引用规则

正式复盘里，买卖理由只能来自三类证据：

1. 飞书原话。
2. 交割单/持仓/委托 RAW。
3. 用户后续确认。

不能来自：

1. Codex 猜测。
2. 低风险识别字段。
3. 市场数据反推。
4. 事后走势倒推。

## 通知链路

脚本生成回执到：

```text
.system/feishu-notify-pending/
```

15分钟任务或手工命令发送：

```bash
python3 .system/scripts/send-feishu-pending-notifications.py
```

发送后归档到：

```text
.system/feishu-notify-sent/
```

当前本机已有飞书 chat_id 记录：

```text
.system/feishu-notifier-state.json
```

## 一句话

飞书口述不是让 AI 猜意思；飞书口述是先把用户原话变成可追溯证据，再由 Codex 在证据基础上复盘。
