# 管道脚本设计原则

基于 `go.txt` 批量转换脚本总结的数据摄入管道设计原则，适用于所有数据清洗与转换工具。

## 原则清单

1. **幂等性与增量处理**
   - 通过 `if (Test-Path $mdPath) { continue }` 实现跳过已存在的输出，确保重复运行安全且高效。
   - 意义：管道可持续运行，无需全量重建。

2. **来源追溯性**
   - 从输入数据中自动提取元数据（如公众号来源行），写入输出头部，使下游可追溯原始出处。
   - 示例：`> 来源: 作者|公众号名` 自动嵌入 MD 文件。

3. **输入清洗优先顺序**
   - 先移除平台专用噪音（如 `<mp-common-profile>`），再处理通用 HTML 标签，最后解码实体。
   - 避免清洗逻辑破坏有意义的内容。

4. **输出格式标准化**
   - 统一输出为带 front-matter 风格头部的 Markdown，便于后续 LLM 解析。

5. **可观测性与进度反馈**
   - 提供文件总数、实时进度报告（每 100 份输出一次）和最终汇总统计（成功/跳过/错误），无需持久化日志即可快速感知执行状态。
   - 未来可扩展持久化日志与告警。

6. **容错与错误隔离**
   - `try-catch` 包裹每个文件，单个文件失败不影响批量任务，错误计数并在最后汇总。

## 反模式警示

- **缺乏日志持久化**：脚本当前仅控制台输出，运行历史不可追溯，不利于长期运维审计。
- **格式假设脆弱性**：正则匹配来源行依赖固定格式，平台变更可能导致来源丢失，需定期验证。
